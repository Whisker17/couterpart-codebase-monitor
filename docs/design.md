## Problem Statement

Tracking the engineering direction of interesting open-source projects is valuable but time-consuming. You want to understand not just what PRs were merged, but what they mean — is a project shifting architectures? Adopting new patterns? The current options are either noise machines (GitHub Watch notifications) or shallow (RSS-style changelogs). None offer opinionated technical narrative that a mixed audience of engineers and strategy people can act on.

## What Makes This Cool

This is an **engineering intelligence agent** — not a changelog summarizer. It reads raw code diffs, uses LLM analysis to infer blast radius and produce directional judgments ("Project X is migrating from monolith to microservices, evidenced by 3 PRs this week introducing service boundaries"). The layered report format serves both the engineer who wants diff details and the strategy person who wants the "so what." Post-MVP, code-review-graph adds AST-level structural analysis for even deeper blast-radius understanding.

The killer feature path (post-v1): **narrative drift detection** — comparing what a project says (README, blog, release notes) with what the code actually shows. And **cross-project trend detection** — when multiple tracked projects converge on the same technology shift. Both are evolution targets, not v1 scope.

## Constraints

- First time using pi-agent; learning curve is real
- Mixed audience requires layered report format (summary + technical detail)
- Lark webhook message size limits (~30KB per message)
- GitHub API rate limit: 5000 requests/hour with token
- LLM cost per PR analysis (need to manage token budget — diff-aware analysis uses ~6000-10000 input tokens/PR)

## Premises

1. pi-agent for LLM tool registration and lifecycle hooks only — pipeline orchestration uses plain TS modules with direct function calls (per eng review)
2. Diff-aware LLM analysis for MVP — raw diffs fed directly to LLM with intelligent truncation. code-review-graph deferred to post-MVP M3 for AST-level blast-radius analysis
3. Daily + weekly reports in M1 (diff-aware analysis from day one), monthly deferred to post-v1
4. Lark webhook as primary delivery channel
5. No local cloning in MVP — all data from GitHub API. Local clones deferred to post-MVP (prerequisite for code-review-graph)

## Cross-Model Perspective

Claude subagent cold read — key takeaways:

- **Coolest version**: A "divergence detector" that scores the gap between a project's marketing narrative (README, blog, release notes) and what the code actually shows. When 73% of merged PRs are fixing single-tenant SQLite bugs despite claims of "production-ready multi-cloud support," that's a HIGH narrative-reality gap. Cross-project trend detection is the second layer: when 3 tracked projects all start adding gRPC in the same month, that's a trend signal.

- **What excites you most**: The word "opinionated" — you want an analyst with a point of view, not a changelog printer. The choice of code-review-graph confirms this.

- **50% tool**: git-cliff + conventional commit parsing gets you activity ingestion and structured changelogs. The hard 50% is the LLM opinion layer, cross-project intelligence, project Overview context, and temporal narrative tracking.

## Approaches Considered

### Approach A: "Ship First" — Minimal viable, layer later
Week 1: GitHub API polling + LLM analysis of PR body/diff stats + Lark push. No local clone, no code-review-graph. Week 2: add code-review-graph. Week 3: add weekly/monthly reports.
- Effort: M (~2 weeks)
- Pro: First report in 1 week, validates prompt and format fast
- Con: Shallow analysis initially, needs two refactors

### Approach B: "Full Stack" — Complete architecture, phased delivery (CHOSEN, modified)
Four pi-agent extensions: GitHub Collector, Diff-Aware Analyzer, Report Generator, Lark Dispatcher. Architecture is designed complete from day one with diff-aware analysis from M1, code-review-graph deferred to post-MVP M3. Delivery: M0 (scaffold), M1 (daily + weekly reports with diff-aware analysis), M2 (harden).
- Effort: L (~3 weeks including learning and polish)
- Pro: Clean architecture, diff-aware analysis from day one, CRG adds as extension later
- Con: Pi-agent learning curve; no AST-level analysis until post-MVP

### Approach C: "Intelligence Layer" — Narrative intelligence first
Project profiles + narrative drift detection + cross-project trends as the core, with PR analysis feeding into profile updates rather than being the primary output.
- Effort: XL (~5 weeks)
- Pro: Strongest version of "align development and narrative changes," unique value
- Con: Needs vector storage, temporal analysis, a month to first report

## Recommended Approach

**Approach B: Full Stack (modified)**, with diff-aware LLM analysis from day one and a clear path to evolve toward C's intelligence features via pi-agent extensions. code-review-graph deferred to post-MVP M3.

### Data Model (SQLite)

```sql
-- Tracked projects
CREATE TABLE projects (
  id TEXT PRIMARY KEY,              -- format: "org/repo"
  org TEXT NOT NULL,
  repo TEXT NOT NULL,
  url TEXT NOT NULL,
  description TEXT,                 -- GitHub repo description
  language TEXT,                    -- GitHub primary language
  topics TEXT,                      -- GitHub repo topics (JSON array)
  overview TEXT,                    -- LLM-generated project profile (post-MVP)
  tech_stack TEXT,                  -- detected technologies (JSON, post-MVP)
  clone_path TEXT,                  -- local path in ./repos/ (post-MVP)
  last_synced_at INTEGER,
  created_at INTEGER DEFAULT (unixepoch())
);

-- Fetched PRs
CREATE TABLE pull_requests (
  id INTEGER PRIMARY KEY AUTOINCREMENT,  -- surrogate key (NOT GitHub PR ID)
  project_id TEXT REFERENCES projects(id),
  pr_number INTEGER NOT NULL,
  github_node_id TEXT,             -- GitHub's global node ID for API reference
  title TEXT NOT NULL,
  body TEXT,
  author TEXT,
  merged_at INTEGER,
  files_changed INTEGER,
  additions INTEGER,
  deletions INTEGER,
  diff_path TEXT,                   -- filesystem path to stored diff (not inline)
  diff_status TEXT CHECK(diff_status IN ('available', 'missing', 'fetch_failed', 'too_large')) DEFAULT 'missing',
  analysis_status TEXT CHECK(analysis_status IN ('pending', 'complete', 'failed')) DEFAULT 'pending',
  retry_count INTEGER DEFAULT 0,     -- analysis retry count (permanent failed after limit)
  last_error TEXT,                   -- most recent failure error message
  fetched_at INTEGER DEFAULT (unixepoch()),
  UNIQUE(project_id, pr_number)
);

-- LLM analyses
CREATE TABLE analyses (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  pr_id INTEGER REFERENCES pull_requests(id),
  project_id TEXT REFERENCES projects(id),  -- denormalized for direct queries
  summary TEXT NOT NULL,           -- 1-2 sentence summary
  technical_detail TEXT,           -- detailed technical analysis
  direction_signal TEXT,           -- what this means for project direction
  significance TEXT CHECK(significance IN ('routine', 'notable', 'directional_shift')),
  categories TEXT,                 -- JSON array: ["architecture", "performance", ...]
  model_id TEXT,                   -- actual model used (e.g. "claude-sonnet-4-6")
  input_tokens INTEGER,            -- prompt tokens consumed
  output_tokens INTEGER,           -- completion tokens consumed
  estimated_cost_usd REAL,         -- estimated cost based on model pricing
  analyzed_at INTEGER DEFAULT (unixepoch())
);

-- Generated reports
CREATE TABLE reports (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  type TEXT CHECK(type IN ('daily', 'weekly', 'monthly')),
  period_start INTEGER NOT NULL,   -- report period start (Unix timestamp)
  period_end INTEGER NOT NULL,     -- report period end (Unix timestamp)
  project_ids TEXT,                -- JSON array of project IDs covered
  content TEXT NOT NULL,           -- formatted report content (Lark card JSON)
  completeness TEXT,               -- JSON: {"total": 5, "success": 4, "failed": ["org/repo"]}
  sent_at INTEGER,                 -- set when all deliveries succeed
  created_at INTEGER DEFAULT (unixepoch()),
  UNIQUE(type, period_start, period_end)
);

-- Per-card delivery tracking (supports split cards for large reports)
CREATE TABLE report_deliveries (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  report_id INTEGER NOT NULL REFERENCES reports(id),
  card_index INTEGER NOT NULL,     -- card sequence (0 for single-card reports)
  content TEXT NOT NULL,           -- single card JSON
  lark_message_id TEXT,            -- Lark response message_id
  status TEXT CHECK(status IN ('pending', 'sent', 'failed')) DEFAULT 'pending',
  sent_at INTEGER,
  UNIQUE(report_id, card_index)
);

-- Analysis input snapshots (persisted at analysis time for audit/replay)
CREATE TABLE analysis_inputs (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  analysis_id INTEGER NOT NULL REFERENCES analyses(id),
  prompt_version TEXT NOT NULL,    -- hash or version tag of prompt template
  input_quality TEXT NOT NULL,     -- "diff_aware" | "metadata_only" | "diff_plus_graph" (post-MVP)
  rendered_project_context TEXT,   -- exact PROJECT CONTEXT block sent to LLM
  file_manifest TEXT,              -- JSON array of FileEntry objects
  diff_included_files INTEGER,
  diff_total_files INTEGER,
  diff_truncated BOOLEAN NOT NULL,
  truncated_diff_path TEXT,        -- path to stored truncated diff snapshot
  created_at INTEGER DEFAULT (unixepoch())
);
```

**Data retention policy:**
- PR diffs (`diff_path` files): retain 30 days after analysis is complete (needed for prompt tuning A/B comparisons), then delete
- Analysis input snapshots (`data/analysis-inputs/*.diff`): retain 30 days, then delete. Metadata rows in `analysis_inputs` table kept indefinitely (small, useful for quality tracking)
- Raw PR body: keep indefinitely (small, useful for re-analysis)
- Analyses: keep indefinitely (the core value)
- Reports: keep 90 days, then archive (export to JSONL files in `data/archive/YYYY-MM/`, then DELETE from SQLite; not queryable without reimport)
- SQLite VACUUM monthly via scheduler

### pi-agent Extension Model

**Critical architectural constraint:** pi-agent's extension system is for LLM tool registration and lifecycle hooks **only**. Pipeline orchestration uses plain TS modules with direct function calls via `src/pipeline/runner.ts` — **not** pi-agent's event system.

Pi-agent extensions are TypeScript files in `src/extensions/` that export a `register(agent: AgentSession)` function. Extensions register interactive tools (e.g. `analyze-pr` for ad-hoc use) and lifecycle hooks, but they do **not** orchestrate pipeline stages. The sequential pipeline (Collector → Analyzer → Report Generator → Dispatcher) is driven by `src/pipeline/runner.ts` calling each stage's `execute()` function directly. Stages communicate via the shared SQLite database (status columns: `pending`/`complete`/`failed`), not events or direct function calls between extensions. Hot-reload means you can modify an extension and run `/reload` without restarting the process. See pi.dev docs: "Extensions" section.

### Diff Truncation Strategy

The `diff-truncator.ts` module prepares raw PR diffs for the LLM prompt within a configurable token budget (default: 8000 tokens / ~32000 chars).

**File priority tiers:**
1. **Skip always**: lock files (`package-lock.json`, `bun.lockb`, `yarn.lock`, `Cargo.lock`, `go.sum`), generated code (`*.generated.*`, `*.min.js`), binary files
2. **Tier 1 — Signal files**: files carrying architectural/directional signals — `package.json`, `Cargo.toml`, `go.mod`, `pyproject.toml`, `Dockerfile`, `docker-compose.*`, `*.proto`, `*.graphql`, `openapi.*`, CI configs (`.github/workflows/*`), migration files (`migrations/*`), Kubernetes manifests
3. **Tier 2 — Source code**: application source files
4. **Tier 3 — Tests**: test files (`*test*`, `*spec*`, `__tests__/`)
5. **Tier 4 — Docs & config**: documentation, linter/formatter config

Within each tier, files are sorted by change size descending. Files are included in tier order until the token budget is reached.

**File manifest:** Always appended to the prompt in compact format (`path +N/-M [included|omitted: reason]`). If >100 files (configurable via `settings.llm.maxManifestEntries`), show top 50 individually, aggregate rest by tier.

**Edge cases:**
- Empty diff → `"(empty diff)"`
- Single large file exceeding budget → include first N lines, note truncation
- >2MB diff → collector sets `diff_status = 'too_large'`, analyzer falls back to metadata-only

### LLM Analysis Specification

**System prompt skeleton:**
```
You are an engineering intelligence analyst. Given a PR and its project context,
produce a structured analysis. Analyze the diff content to understand the actual
code changes, their patterns, and what they suggest about the project's
engineering direction.

PROJECT CONTEXT:
{projectContextLite — description, language, topics, tags, notes}

PR INFORMATION:
Title: {pr.title}
Author: {pr.author}
Files changed: {pr.files_changed} (+{pr.additions}/-{pr.deletions})
PR Body: {pr.body (truncated to 1000 chars)}

DIFF CONTENT:
{truncated_diff.content or "Diff not available — analysis based on PR metadata only."}
{if truncated: "(Diff truncated: showing N/M files within token budget)"}
{fileManifest in compact format}

SUPPLEMENTARY CONTEXT:
{analysisContext.supplementaryContext or "Not available."}

Respond in JSON:
```

**Required output schema:**
```json
{
  "summary": "1-2 sentences, what this PR does and why it matters",
  "technical_detail": "2-4 sentences, key technical changes and their implications",
  "direction_signal": "1 sentence, what this suggests about the project's direction. null if routine.",
  "significance": "routine | notable | directional_shift",
  "categories": ["architecture", "dependency", "api", "performance", "security", "testing", "docs"]
}
```

**Significance classification rubric:**
- `routine`: Bug fixes, minor refactors, test additions, dependency bumps, doc updates
- `notable`: New features, significant refactors (>10 files), new dependency categories, performance changes with benchmarks
- `directional_shift`: New architectural patterns (e.g., adding gRPC to a REST-only project), language/framework migrations, major API surface changes (>5 endpoints), new infrastructure patterns (e.g., adding Kubernetes manifests to a bare-metal project)

### Lark Card Implementation

Use Lark's **Message Card v2** API (`POST /open-apis/im/v1/messages` with `msg_type: "interactive"`). Card JSON schema reference: https://open.larksuite.com/document/common-capabilities/message-card/getting-started/card-structure/card-content

For the mixed-audience layered format, use `column_set` with a visible summary section and a collapsible `action` element for technical details:

```json
{
  "config": { "wide_screen_mode": true },
  "header": { "title": { "tag": "plain_text", "content": "Counterpart Monitor · Daily Digest · 2026-05-27" } },
  "elements": [
    { "tag": "markdown", "content": "**Summary**\n..." },
    { "tag": "hr" },
    { "tag": "collapsible_panel", "expanded": false,
      "header": { "title": { "tag": "plain_text", "content": "Technical Details" } },
      "elements": [{ "tag": "markdown", "content": "..." }]
    }
  ]
}
```

**Message size limit strategy:** Lark cards have a ~30KB limit. If a daily report exceeds 20KB:
1. Include only `notable` and `directional_shift` PRs in the card
2. Append a "N routine PRs omitted" line
3. If still over limit, split into one card per project

### Cost Model

Estimated token usage per PR analysis (diff-aware):
- Input: ~6,000-10,000 tokens (system prompt ~300 + project context lite ~200 + PR metadata ~200 + truncated diff ~5,000-8,000 + file manifest ~300)
- Output: ~300-400 tokens (structured JSON)
- Model: Claude Sonnet 4 ($3/M input, $15/M output as of May 2026). Costs vary by the Anthropic-compatible gateway and model pricing used in settings.
- Cost per PR: ~$0.02-0.04

Monthly estimate for 10 repos, ~5 PRs/day average:
- 50 PRs/day x 30 days = 1,500 analyses/month
- Cost: ~$30-60/month
- Weekly/monthly report generation: ~$2-5/month (fewer calls, larger context)
- **Total: ~$35-65/month**

Budget controls:
- Track `input_tokens`, `output_tokens`, `model_id`, `estimated_cost_usd` in analyses table
- **M1 (hard cap):** Monthly budget cap configurable in settings (default: $80). When reached, analyzer stops and remaining PRs stay `pending`. StageResult records `budgetExhausted: true`, daily report includes budget warning.
- **M2 (fine-grained, Issue 22):** When 80% reached: pre-filter to skip likely-routine PRs (<5 files AND <100 additions). When 100%: pause analysis, send alert to Lark. Budget dashboard in daily report.

### Failure Modes

| Failure | Detection | Recovery |
|---------|-----------|----------|
| GitHub API 403 (rate limit) | HTTP status | Exponential backoff, retry after `X-RateLimit-Reset` header |
| GitHub API 404 (repo deleted/renamed) | HTTP status | Mark project inactive, log warning to Lark |
| LLM API timeout/error | HTTP status or timeout >60s | Retry 2x with backoff, then mark `analysis_status = 'failed'` + increment `retry_count`, auto-retry next run |
| Lark webhook failure | HTTP status != 200 | Retry 3x, then mark `report_deliveries.status = 'failed'`, report stays `sent_at IS NULL`, auto-retry next run |
| Budget cap reached | Monthly cost estimate > cap | Stop analysis, remaining PRs stay `pending`, report includes warning |
| SQLite lock contention | SQLITE_BUSY | WAL mode enabled by default, 5s busy timeout |

Pipeline runs sequentially (collect → analyze → report → dispatch), so a failure in one stage does not corrupt the next. Failed items are marked in DB and retried on next run.
