## Key Architectural Insights

### 1. pi-agent is not for pipelines (confidence: 8/10, cross-model)

pi-agent extension system is designed for LLM tool registration and lifecycle hooks, not batch data pipeline orchestration. For sequential pipelines (fetch -> analyze -> report -> dispatch), use plain TS modules with direct function calls and reserve pi-agent for the LLM analysis step only.

**Files:** `src/pipeline/runner.ts`, `src/index.ts`

### 2. Git shallow clone fetch depth pitfall (confidence: 9/10, observed)

> **Deferred to post-MVP M3** — CRG removed from MVP scope per approved spec (`docs/spark/2026-05-29-remove-crg-from-mvp-design.md`). No local clones in MVP.

Using `git fetch --depth=N` on an existing shallow clone truncates visible history to N commits from tip, not extends it. This breaks `git merge` when merge base is outside the shallow boundary. Use `git fetch origin` (no `--depth`) for incremental updates to shallow clones.

**Files:** `src/pipeline/collector.ts`

### 3. bun:sqlite production pragma suite (confidence: 9/10, observed)

bun:sqlite production setup needs more than WAL mode. Add:
- `temp_store=MEMORY`
- `cache_size=-64000`
- `mmap_size=268435456`

On macOS, WAL sidecar files persist after close. Use `fileControl(SQLITE_FCNTL_PERSIST_WAL, 0)` + `wal_checkpoint(TRUNCATE)` for clean shutdown.

**Files:** `src/storage/db.ts`

### 4. SQLite as inter-extension communication layer (confidence: 7/10, inferred)

For multi-stage data pipelines in pi-agent, use SQLite as the communication layer between extensions rather than direct function calls. Each stage reads from and writes to DB tables, with status columns (`pending`/`complete`/`failed`) to track progress. This makes the pipeline resumable and debuggable.
